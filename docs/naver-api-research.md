# TrendDrop Naver API Research

작성일: 2026-07-06

## 목적

TrendDrop의 "한국형 소비 트렌드" 포지셔닝의 기둥이 될 네이버 계열 API를 실사하고, 트렌드 발견/검증/맥락 레이어 각각에 무엇을 쓸 수 있는지, 쿼터·비용·약관 리스크는 어떤지 정리한다.

이 문서는 아래 질문에 답하는 것을 목표로 한다.

- 네이버 데이터랩은 트렌드 "발견" 소스인가, "검증" 레이어인가
- 절대 검색량은 어디서 얻을 수 있는가
- 상업 서비스(광고 포함 가능성 있는 소비자 앱)에서 약관상 무엇이 걸리는가
- 쿼터와 갱신 주기는 우리 아키텍처(배치 수집 → DB → 내부 API)에 맞는가

조사 방법: 공식 문서 원본 저장소(GitHub `naver/naver-openapi-guide`, 2026-05-22 최종 갱신 확인), 검색광고 API 공식 Swagger 스펙과 샘플 코드, datalab.naver.com 웹 실측(2026-07-06), 공식 공지 원문을 직접 확인했다. 실제 API 키 발급·인증 호출은 수행하지 않았다(미검증 항목에 명시). 카카오/빅카인즈는 이번 조사에서 제외했다(후속 조사 항목).

## 한눈에 보는 결론

### 핵심 판단

- **데이터랩 API는 트렌드 발견 소스가 될 수 없다 — 확정.** 키워드를 입력해야 상대지수(ratio 0~100)가 나오는 구조이고, "인기 키워드 목록"을 주는 엔드포인트가 없다. 다른 소스(TikTok Discovery 등)에서 발견한 키워드의 **검증·정량화 레이어**로만 쓸 수 있다
- **절대 검색량의 유일한 공식 경로는 검색광고 API 키워드도구**(`/keywordstool`) — 최근 30일 PC/모바일 검색수 절대치 + 연관 키워드 + 경쟁도를 무료로 제공. 개인(비사업자)도 본인인증으로 발급 가능
- **가장 큰 리스크는 기술이 아니라 약관.** 오픈API는 "저장·재가공 원칙 금지 + 광고 병행 노출 금지 + 제3자 재제공 금지"가 명문화되어 있어, 우리의 "수집 → DB 적재 → 소비자 앱 표시" 구조는 **사전 제휴 신청 없이는 문면상 위반 소지**가 있다. 검색광고 API 쪽은 약관 전문이 로그인 뒤에 있어 미확인(발급 시 최우선 확인 사항)
- 데이터랩 웹의 카테고리별 인기검색어 랭킹(발견에 가장 필요한 데이터)은 **API 미제공 + 크롤링 명시 금지**(약관이 정보통신망법 위반 소지까지 언급)
- 실시간 급상승 검색어는 2021-02 폐지 확정, 공식 대체 API 없음 (2023년 "트렌드 토픽"도 실검 부활 논란으로 철회)

### 바로 검토할 가치가 큰 API

1. 검색광고 API 키워드도구 — 절대 검색량 검증 레이어
2. 데이터랩 검색어트렌드 API — 시계열·인구통계 분해 (제휴 협의 전제)
3. 검색 API(뉴스/블로그/카페) — D3 "왜 뜨나" 근거·D6 관련 콘텐츠 (약관 특약 주의)

### 보조

1. 데이터랩 쇼핑인사이트 API — 쇼핑 카테고리 클릭 추이 (카테고리 부분 일치)

### 제외 또는 조건부

1. 데이터랩 웹 인기검색어 크롤링 (약관 명시 금지 — 필요하면 제휴 협의 대상)
2. 네이버 클라우드 Cloud Data Box (유료 기업용, 소규모 스타트업에 비현실적일 가능성)

## API별 조사

## 1. 데이터랩 검색어트렌드 API

공식 문서:

- https://developers.naver.com/docs/serviceapi/datalab/search/datalab.search.md

무엇을 할 수 있나:

- `POST openapi.naver.com/v1/datalab/search` — 키워드 그룹(최대 5개, 그룹당 키워드 20개)의 검색 관심도 시계열 조회
- 기간: 2016-01-01부터, `date`/`week`/`month` 단위
- 필터: 기기(pc/mo), 성별, 연령 11개 구간

결정적 제약:

- **상대 비율만 제공** — "구간별 결과에서 가장 큰 값을 100으로 설정한 상댓값". 절대량 필드 자체가 없음
- 비율 기준(=100)이 **요청 단위 내 최대값**이라 서로 다른 요청 간 ratio 직접 비교 불가 → **앵커 키워드(기준 키워드)를 모든 요청에 공통 삽입해 정규화**하는 설계 필요
- 키워드를 입력해야만 결과가 나옴 → **발견 불가**

쿼터/비용: 1,000회/일, 무료. `X-Naver-Client-Id`/`X-Naver-Client-Secret` 헤더 인증. 요청당 5그룹이므로 하루 최대 5,000키워드-그룹 — 검증 레이어 용도로 충분. 부족 시 제휴 신청으로 무료 증액 경로 있음.

TrendDrop 활용: 발견된 키워드의 **관심도 시계열(급상승 산식 D2의 보조 신호) + 성별/연령 분해(카테고리 감도)**. 단 아래 §5 약관 리스크 참조.

## 2. 데이터랩 쇼핑인사이트 API

공식 문서:

- https://developers.naver.com/docs/serviceapi/datalab/shopping/datalab.shopping.md

무엇을 할 수 있나:

- 엔드포인트 8종: 분야별 트렌드(카테고리 최대 3개), 분야 내 기기/성별/연령별, **분야 내 키워드별 트렌드**(키워드 그룹 최대 5개·그룹당 검색어 1개), 키워드×인구통계
- 데이터: 2017-08-01부터. 반환은 역시 **상대 비율(ratio)만** — 네이버 통합검색 쇼핑영역+네이버쇼핑의 "검색 클릭" 추이

카테고리 매핑 (1차 카테고리, 2026-07-06 웹 실측):

| TrendDrop 카테고리 | 네이버쇼핑 카테고리 (cat_id) | 정합성 |
|---|---|---|
| 뷰티 | 화장품/미용 (50000002) | ⭕ |
| 패션 | 패션의류 (50000000) + 패션잡화 (50000001) | ⭕ |
| 푸드 | 식품 (50000006) | ⭕ |
| 테크 | 디지털/가전 (50000003) | ⭕ |
| 여행 | 직접 대응 없음 (여가/생활편의 하위에 일부) | ❌ |
| 라이프스타일 | 생활/건강 + 가구/인테리어 등에 분산 | 🔶 |

쿼터: 1,000회/일 (검색어트렌드와 별도), 무료.

TrendDrop 활용: 뷰티/패션/푸드/테크 4개 카테고리의 **쇼핑 수요 보조 신호**. 여행·라이프스타일은 커버 안 됨. 인기 키워드 목록은 API로 안 나옴(웹에만 있고 크롤링 금지).

## 3. 검색 API (뉴스/블로그/카페글)

공식 문서:

- https://developers.naver.com/docs/serviceapi/search/news/news.md (블로그/카페글 동일 계열)

무엇을 할 수 있나:

- `GET /v1/search/news.json` 등 — 키워드로 최신/정확도순 문서 검색. `display` 최대 100, `start` 최대 1000
- 뉴스: 제목/원문링크/네이버뉴스링크/요약문/발행일. 블로그·카페도 유사

쿼터: **25,000회/일** (뉴스·블로그·카페 등 검색 API 전체 합산, 클라이언트 아이디 단위). 무료.

TrendDrop 활용: **D3("왜 뜨나" 설명 생성의 근거 문서) + D6(관련 콘텐츠 묶음)**에 기술적으로 적합. 단:

- 약관 특약: 검색결과를 독립적으로 노출해야 하고 앞뒤/중간에 다른 내용 삽입·왜곡·임의 수정 금지. "검색결과를 보여주는 서비스"의 상업적 이용 불가 명시 → **검색결과를 그대로 나열하는 UI는 위험**하고, LLM 요약의 근거 자료로 서버 내부에서만 쓰는 방식도 "재가공 금지" 문면과 충돌 소지 → 제휴 협의 필요
- `total`(총 결과 수)의 시계열 추적을 언급량 프록시로 쓰는 것은 공식 지원 용도가 아님

## 4. 검색광고 API 키워드도구 (절대 검색량)

공식 자료:

- https://naver.github.io/searchad-apidoc/ (공식 Swagger 스펙에서 응답 필드 직접 확인)
- https://github.com/naver/searchad-apidoc

무엇을 할 수 있나:

- `GET api.searchad.naver.com/keywordstool` — 시드 키워드 **최대 5개(콤마 구분)** → 연관 키워드 목록 + 키워드별 지표:
  - `monthlyPcQcCnt` / `monthlyMobileQcCnt`: **최근 30일 검색수 절대치** (10 미만은 `"< 10"` 마스킹)
  - 월평균 클릭수/클릭률, 파워링크 노출 깊이, 경쟁도(`compIdx`: low/mid/high)
- Google Keyword Planner가 구간(band)만 주는 것과 달리 **절대치를 줌** — 오히려 더 정밀
- 연관 키워드가 대량 반환되어 요청 수 대비 커버리지가 좋음 (상한 통념 ~1,000개는 공식 근거 없음)

자격/인증:

- 네이버 검색광고(광고주) 계정 필요 — **광고 집행 없이도 발급 가능, 개인(비사업자)도 본인인증으로 가입 가능** (커뮤니티 근거, 공식 명문 미확인)
- 광고시스템 → 도구 > API 사용관리에서 즉시 발급: CUSTOMER_ID + API KEY + Secret
- 인증: 요청마다 `{timestamp}.{method}.{uri}`를 HMAC-SHA256 서명 (공식 샘플 코드 있음)
- 무료

레이트리밋 (공식 공지 원문 확인):

- 고정 수치 비공개 — "계정당 속도 제한, 전체 사용량에 따라 유연한 형태"
- **키워드도구는 다른 오퍼레이션 대비 1/5~1/6 수준으로 강하게 제한**, IP 기준 제한도 병행
- 429 수신 시 5~6배 긴 sleep 권장 → **저속 배치(수 초당 1회 수준) + 지수 백오프 + 캐싱 필수**

데이터 신선도: "최근 30일" rolling window. 커뮤니티 관찰상 창이 매일 하루씩 이동(사실상 일 단위 갱신, 공식 미확인). 값이 10 단위 반올림 + 소급 보정 노이즈 관찰 → 점수화 시 이동평균 스무딩 권장.

약관 리스크 (이번 조사 최대 공백):

- **API 이용약관 전문이 로그인 후 신청 단계에서만 열람 가능** — 제3자 제공/공개 표시 가부를 공식 확인 못 함
- 생태계 관행: 블랙키위, 키워드마스터 등 다수 서비스가 네이버 월간 검색수를 대중에 공개 표시하며 장기 운영 중 — 다만 이는 "묵인되는 관행"이지 법적 안전판이 아님
- 완화 설계: 원시 검색수 대신 **파생 점수(정규화 수요 지수)**를 표시하고 원시값은 서버 내부 보관 — 단 이것도 허용된다는 공식 근거는 없음

TrendDrop 활용: **검색 수요 검증 레이어의 핵심.** Google 조사가 Keyword Planning에 부여한 역할의 한국판이며 데이터는 더 정밀. 기존 아키텍처 원칙(배치 → DB → 내부 점수 반영 → 화면엔 계산된 점수만)과 정확히 맞물림.

## 5. 오픈API 공통 약관 — TrendDrop에 가장 중요한 제약

출처: 공식 이용약관(2025-05-27 개정) + FAQ

1. **저장·재가공 원칙 금지**: FAQ 원문 — "부하에 따른 단순 캐시 목적이 아니라면 별도로 저장하거나 재가공하여 이용하실 수 없습니다" → **배치 수집 → DB 적재 → 재가공 표시 구조가 문면상 원칙 금지에 해당**
2. **광고 병행 노출 금지**: API 결과 화면에 배너광고/스폰서 링크가 붙거나 유료 회원 전용이면 "상업적 이용" → 금지. 사용자에게 API 이용 대가 수취 금지
3. **제3자 재제공 금지**: TrendDrop 내부 API가 데이터랩 응답을 사실상 그대로 중계하면 저촉 소지
4. **쿼터 우회 금지**: 다수 클라이언트 아이디 분산 호출 금지
5. **크롤링 금지**: 사전 승인 없는 자동화 수단의 주기적/반복적 접속 금지 (정보통신망법 제48조 위반 소지 명시)
6. **표시 의무**: 네이버 오픈API 사용 사실을 링크/로고로 표시
7. **해소 경로**: 제휴 신청 공식 존재 — https://developers.naver.com/openapi/cooperation / dl_naver_openapi@navercorp.com — "제휴를 통한 사용도 무료". **상용화 전 서면 허용 범위 확보가 사실상 전제 조건**

## 설계 시사점 (기존 아키텍처 원칙에 추가)

- **역할 분담 확정**: 발견(TikTok Discovery 등) → 검증(검색광고 키워드도구: 절대량 / 데이터랩: 시계열·인구통계) → 맥락(검색 API 뉴스·블로그) → 임베드(TikTok oEmbed)
- **앵커 키워드 정규화**: 데이터랩 ratio는 요청 간 비교 불가 → 모든 요청에 공통 기준 키워드 삽입
- **수집 주기**: 검색광고 검색수는 일 단위 rolling → 일 1회 배치면 충분. 데이터랩도 일 단위. 실시간 폴링 불필요
- **키워드도구 스로틀링**: 수 초당 1회 + 지수 백오프 + 429 시 5분 중단 로직
- **화면에는 파생 점수만**: 원시 검색수/ratio를 그대로 노출하지 않는 것이 약관·차별화 양면에서 유리
- DB 스키마: `trend_snapshots`에 절대량 numeric 컬럼(monthly_pc_qc, monthly_mo_qc 등) 추가 필요 — 현재 varchar growth/velocity 구조로는 부족 (prior-research-review에서 지적한 사항과 동일)

## 미검증 항목 (명시)

- 검색광고 API **이용약관 전문** (제3자 제공/공개 표시 가부) — 발급 시 최우선 확인
- 키워드도구 연관 키워드 최대 반환 개수 (통념 ~1,000개, 공식 근거 없음)
- 검색광고 API의 구체적 호출 한도 수치 (공식적으로 "유동적"이라고만 명시)
- 검색수 "매일 갱신"은 커뮤니티 관찰 (공식 확인 없음)
- 개인(비사업자) 가입 절차 세부·광고 미집행 계정의 라이선스 제한 유무
- 쇼핑인사이트 면세점/도서 카테고리의 API 실조회 가능 여부
- 검색 API 25,000회/일이 "전체 합산"이라는 해석 (문서 문구 기반 도출, 공식 명문 없음)
- 데이터랩 API 데이터 갱신 지연(당일 데이터 제공 여부)
- Cloud Data Box의 절대 검색량 제공 범위·가격
- 제휴 신청의 승인 기준·처리 기간·증액 상한
- 실제 인증 호출은 이번 조사에서 미수행 — 모든 응답 스키마는 문서/스펙 기준
