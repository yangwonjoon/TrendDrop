## 총평

**전체적으로 수준 높은 사전조사다.** 조사가 조사로 끝나지 않고 의사결정 안건(체크박스)과 코드 스키마까지 이어지는 흐름이 잘 설계되어 있고, "공식 API 부재"를 근거로 판단하는 태도(Google Trends 사례)나 과거 실패 경험을 자산화한 방식은 그대로 팀 표준으로 삼을 만하다.

**리뷰 범위에 대한 전제**: 이 작업자의 담당은 **Google 계열 조사**였다. 따라서 Meta(인스타/페북), TikTok, X 등 타 플랫폼 조사가 없는 것은 이 작업자의 방법론 문제가 아니라 **프로젝트 차원에서 아직 남아 있는 조사 영역**이다(문서 말미에 별도 정리). 아래 피드백은 담당 범위 안에서의 조사 품질에 집중한다 — 핵심 보완점은 실호출 검증(POC) 부재와 쿼터 수치의 정확성 두 가지다.

## 잘한 점 (방법론)

### 1. 결론 먼저, 일관된 문서 구조

모든 문서가 `목적 → 한눈에 보는 결론 → 상세 → 시사점 → 참고 링크` 구조를 지킨다. 서비스별 조사도 동일 프레임(무엇을 하나/특징/장점/한계/시사점)을 반복해서 서비스 간 비교가 가능하다. 읽는 사람 기준으로 쓰인 문서다.

### 2. "부재의 확인"을 근거로 삼는 태도

google-api-research의 백미. Google Trends 공식 API가 "없다"는 것을 확인하고, 그 부재 자체를 "핵심 전제로 잡지 말 것"이라는 판단 근거로 명시했다(`이 부분은 공식 문서 부재를 근거로 한 판단이다`). 비공식 크롤링의 리스크까지 짚었다. 흔히 놓치는 리서치 규율이다.

### 3. 조사 → 결정거리 변환

trend-services-research의 결론이 domain-differentiators에서 우선순위 색상(🔴🟠🟡🟢)과 체크박스가 달린 결정 안건(D1~D8)으로 변환됐다. "내일 논의 핵심 3가지"로 압축한 것, D3+D6(왜 뜨나 + 근거 콘텐츠)를 해자로 지목하고 D1(데이터 소스)이 최대 난제임을 스스로 인정한 것 모두 정확한 자기 인식이다.

### 4. 실패 경험의 자산화

issues-from-sticker-trend-app은 과거 프로젝트의 사고 5종(무한 로딩류), 보안 구멍, 미사용 AI 라우트 같은 실수를 심각도별 이슈 + 재사용 가능 자산 표로 정리했다. "완료 조건"을 각 이슈에 명시한 것도 좋다.

### 5. 아키텍처 원칙이 구체적이고 옳음

"사용자 요청마다 외부 API 직접 호출 금지, 배치 수집 → DB → 내부 API", 소스별 freshness 차등, stale cache fallback, 쿼터 예산 역산 — 이 원칙들은 TikTok 조사에서도 그대로 유효했고, 실제 파운데이션 코드 구조(DB 폴백)와도 일치한다.

## 아쉬운 점 (담당 범위 안에서의 방법론 공백)

### 1. 🟠 탁상 조사만 있고 실호출 검증(POC)이 없음

google-api-research는 문서 기반 조사로는 훌륭하지만, 실제 API 키를 만들어 한 번이라도 호출해 본 흔적이 없다. 쿼터 구조, 응답 형태, 승인 절차의 실제 마찰은 문서만으로 확인되지 않는다. sticker-app 교훈 #7("틀만 만들고 방치 금지 — 실제 호출까지가 완료")과 같은 원칙이 조사에도 적용됐어야 한다.

### 2. 🟠 수치 서술의 정확성 — YouTube 쿼터 재확인 필요

google-api-research의 `기본 할당량은 search.list 100회/일, videos.insert 100회/일, 그리고 나머지 엔드포인트 합산 10,000 units/day` 서술은 공식 쿼터 구조와 다르게 읽힐 소지가 있다. 공식 구조는 **프로젝트당 일 10,000 유닛 통합 예산이고, search.list가 호출당 100유닛을 차감**하는 방식이다(그래서 검색만 쓰면 사실상 일 100회). 별도 엔드포인트별 일일 횟수 제한처럼 서술하면 예산 설계가 어긋난다. 재확인 필요.

또한 **`videos.list chart=mostPopular`(지역별 인기 동영상 차트, 호출당 1유닛) 누락**이 아쉽다. 트렌드 서비스 입장에서 search.list(100유닛)보다 100배 싸고 목적에도 더 맞는 엔드포인트인데 조사에 없다. YouTube 연동 시 1순위로 검토할 것.

### 3. 🟡 비용 역산 부재

"쿼터 예산을 먼저 잡고 기능을 설계하라"고 원칙을 세워놓고, 정작 제시한 카덴스(6카테고리 × 20키워드 × 30분 주기)의 호출량/유닛/비용 역산 표가 없다. TikTok 조사에서 이 역산을 해보니 과금 모델에 따라 **월 $86 vs $5,900**까지 갈리는 게 확인됐다 — 이 계산이 소스 선정 자체를 뒤집을 수 있으므로 조사 단계에서 반드시 포함되어야 한다.

### 4. 🟡 의사결정 기록 부재

두 이슈 문서 모두 "내일 팀 논의용"인데, 그 논의의 **결론이 어디에도 기록되어 있지 않다.** 스택이 Firebase 가정에서 Neon+Drizzle로 바뀐 것은 코드로만 암시된다. D1~D8의 체크박스도 전부 미체크 상태다. 결정이 났으면 문서에 반영하고, 안 났으면 그 사실을 남겨야 다음 사람(지금의 나)이 추측하지 않는다.

### 5. 🟡 검증 상태 표기의 일관성

google-api-research는 부분적으로 검증 근거를 밝히지만, 문서 전반에서 "공식 문서로 확인한 사실"과 "추정/해석"의 구분 표기가 일관되지 않다. 쿼터·가격처럼 변동성 높은 수치는 확인 일자를 주장 단위로 남겨야 한다.

## 코드 반영 상태 리뷰 (조사 → 코드)

조사와 별개로, 파운데이션 코드가 조사 결론을 얼마나 따르는지도 짚는다.

**잘 반영된 것:**

- [schema.ts](../db/schema.ts)의 `summary`/`reason`/`sourceLabel` 필드는 D3(왜 뜨나)·D6(감지 채널)의 씨앗으로 정확히 대응됨
- DB 미설정 시 목데이터 폴백([trends-service.ts](../lib/trends-service.ts)) — 이슈 #4(장애 시 폴백)의 정신과 일치
- 조사 문서를 앱 내 뷰어(`/docs`)로 노출한 것 — 조사 산출물의 접근성을 높인 좋은 선택

**어긋나거나 보완 필요한 것:**

- **`growthRate`/`velocity`가 varchar** (`"+182%"`, `"9.1/10"` 표시 문자열 그대로 저장). D2가 요구하는 "설명 가능한 산식"을 하려면 원시 수치(조회수, 게시물 수, 일별 시계열)를 numeric으로 저장하고 표시값은 계산해야 한다. TikTok Discovery API의 `trending_history`(일별 순위/조회수)를 받으려면 스키마 확장이 필요하다
- **`/api/admin/bootstrap` POST가 무인증** — 이슈 #1의 교훈("보안 규칙은 0일차에")이 정작 이 브랜치에 적용되지 않았다. 배포되면 누구나 시드를 트리거할 수 있다. 관리용 시크릿 헤더라도 먼저 달아야 한다
- **소스 표기의 허구성**: `sources`에 "YouTube Data API"가 시드되지만 수집기는 없고, 목데이터의 `sourceLabel`은 "Instagram Reels, Facebook Groups" 등이다. P1-5(데이터 소스 표시)가 "사용자 신뢰"를 위한 기능인 만큼, 실소스 연결 전까지는 목데이터임을 UI에 명시하는 편이 안전하다 (D7의 "마지막 갱신 시각 노출"과 같은 맥락)

## 개선 제안 — 다음 조사부터 적용할 규율

1. **조사 문서 템플릿에 "조사 방법"과 "미검증 항목" 섹션 의무화** — 무엇을 직접 확인했고 무엇이 추정인지 분리 (tiktok-api-research.md에 적용해 둠)
2. **API 조사의 완료 조건 = 실호출 1회 이상** — 키 발급/승인 마찰, 실제 응답 형태까지 확인해야 "조사 완료"
3. **비용 역산 표 필수** — 제품 카덴스(카테고리 × 키워드 × 주기) 기준으로 호출량과 월 비용을 계산. 과금 모델이 소스 선정을 뒤집을 수 있음
4. **결정 기록(decision log) 도입** — 회의에서 정해진 것을 해당 문서의 체크박스에 반영하거나 `docs/decisions.md`에 한 줄씩 누적
5. **변동성 수치에 확인 일자 병기** — 쿼터/가격/약관은 주장 단위로 "2026-07 확인" 태그
