# trendDrop 도메인 차별점 이슈 초안

> 작성일: 2026-06-24 · 목적: 내일 팀 논의용 (엔지니어링 이슈는 [issues-from-sticker-trend-app.md](issues-from-sticker-trend-app.md) 참고)
>
> [trend-services-research.md](trend-services-research.md)에서 정한 차별화 방향을 **"풀어야 할 도메인 문제 + 결정거리"**로 옮긴 것.
> 앞 문서(#1~#9)가 *어떻게 잘 만들까*(공통)였다면, 이 문서는 *trendDrop을 왜 다르게 만드는가*(고유)에 대한 것.

## 한 줄 정의 (리서치 결론 재확인)
> Google Trends가 "무엇이 **검색**되는가"라면, trendDrop은 "사람들이 SNS에서 **무엇에 반응하고 왜 퍼뜨리는가**"를 보여준다.

## 차별화 해자(moat) 우선순위 — 내일 합의할 핵심
리서치상 진짜 방어 가능한 차별점은 **"왜 뜨나" 설명(D3) + 관련 콘텐츠 묶음(D6)**. 단순 랭킹(누가나 만든다)이 아니라 *맥락*이 해자.
→ **D3·D6를 MVP에서 무엇으로 증명할지**가 가장 중요한 논의. 나머지(D1·D2)는 그걸 받치는 토대.

---

## D1. 🔴 소셜 "반응" 신호 수집 파이프라인 — 차별점의 근간이자 최대 난제

**왜 차별점인가**: trendDrop의 정체성은 *검색량*이 아니라 *저장·공유·댓글 같은 SNS 반응*. 이 데이터를 못 얻으면 그냥 또 하나의 검색 트렌드 클론.

**풀어야 할 문제**
- 인스타그램/메타 공식 API는 트렌드/타인 게시물 반응 접근이 강하게 제한됨. 스크래핑은 합법성·안정성 리스크.
- 어디서 시작할 것인가: ① 메타 Graph API(가능 범위 한정) ② 공개 해시태그/커뮤니티 ③ 뉴스·검색 보조(네이버 데이터랩 등) ④ **반자동 큐레이션 부트스트랩**.

**결정 필요**
- [ ] 1차 데이터 소스 1~2개 확정 (현실적으로 얻을 수 있는 것부터)
- [ ] 초기엔 큐레이션/반자동으로 부트스트랩하고 점진 자동화할지 (Trend Hunter 모델) — *완전 자동 실시간*을 처음부터 노릴지 여부
- [ ] 수집 합법성/ToS 검토 책임자

**MVP 최소선**: "반응 신호"라고 부를 수 있는 단 하나의 실데이터 소스라도 붙여서 하드코딩 목 데이터를 대체.

---

## D2. 🟠 "급상승" 판정 로직 — growth / velocity를 어떻게 정의할 것인가

**왜 차별점인가**: 현재 프로토타입의 `+182%`, `확산속도 9.1/10`은 [app.js](../app.js)에 **하드코딩**. trendDrop이 "급상승 감지"를 표방하는 이상, 이 숫자의 *정의*가 제품 신뢰의 핵심.

**풀어야 할 문제**
- 상승률 산식: 무엇 대비(baseline)? 시간 윈도우(오늘/24h/7d/30d)?
- "확산 속도" 점수화: 단순 증가량 vs 가속도(2차 미분).
- 노이즈 제거: 봇·일시적 급등·계절성 vs 지속형 트렌드 구분.

**결정 필요**
- [ ] growth/velocity의 최소 정의(v1) 합의 — 정교한 모델 전에 *설명 가능한 단순 산식*부터
- [ ] 기간 필터(오늘/24h/7d/30d) 제공 범위 (리서치 P2 항목)

---

## D3. 🔴 "왜 뜨나" 설명 레이어 — 최대 차별점 / 해자

**왜 차별점인가**: 리서치 결론 = 단순 랭킹은 금방 대체됨. **키워드 + 반응 이유 + 관련 콘텐츠 묶음**이 진짜 해자. Trend Hunter/BuzzSumo 시사점도 동일("숫자만 말고 맥락을").

**sticker-app 경험 직결**: AI 요약 서버 라우트 패턴(`api/ai-summary`)을 **그대로 재사용** 가능. 도메인만 "스티커 레퍼런스 요약" → "트렌드 맥락 설명"으로 교체. (구현 패턴·주의점은 엔지니어링 이슈 #7)

**풀어야 할 문제 / 결정 필요**
- [ ] 설명 생성의 입력: 관련 게시물·뉴스·연관 키워드 중 무엇을 근거로 줄 것인가
- [ ] **환각 방지**: 출처 인용/근거 게시물 링크를 함께 노출 (D6와 결합)
- [ ] 생성 비용·신선도: 트렌드마다 매번 LLM 호출 vs 캐싱/배치

**MVP 최소선**: 트렌드 카드의 "왜 뜨나" 한 줄이 실제 근거(관련 게시물/뉴스)에 연결된 AI 요약으로 채워짐.

---

## D4. 🟠 예비 급상승(조기 탐지) — "남들보다 먼저 본다"

**왜 차별점인가**: 이미 뜬 것만 보여주면 늦음. *곧 뜰 것*을 분리 제공 = Exploding Topics식 가치. 현재 프로토타입의 "예비 급상승 키워드" 워치리스트가 자리는 잡혀 있음.

**풀어야 할 문제 / 결정 필요**
- [ ] "지금 뜨는 것"과 "곧 뜰 것"의 판정 경계
- [ ] 조기 신호 오탐 허용도 (너무 예측 위주면 체감 가치 약화 — 리서치 경고)

---

## D5. 🟡 한국형 소비 카테고리 + 자동 분류

**왜 차별점인가**: 해외 서비스는 글로벌 검색/리포트형. trendDrop은 한국 SNS 소비·쇼핑 전 탐색 흐름. 초기 카테고리: 뷰티/패션/푸드/여행/라이프/테크 (리서치 확정).

**결정 필요**
- [ ] 키워드 → 카테고리 자동 태깅 방식 (규칙 기반 vs LLM 분류)
- [ ] 카테고리별 피드 + 전체 피드 구조 (YouTube Charts 시사점: 한 줄 나열보다 카테고리 차트)

---

## D6. 🔴 관련 콘텐츠/출처 묶음 + 감지 채널 신뢰 — D3와 한 쌍

**왜 차별점인가**: "왜 뜨나"를 *증명*하는 근거. 대표 게시물/기사를 묶어 보여주면 BuzzSumo식 강점 + 설명의 신뢰 확보. 현재 카드의 "감지 채널" 표시가 그 씨앗.

**sticker-app 경험 직결**: 레퍼런스(링크·이미지·플랫폼·태그) 수집·저장 데이터 모델이 여기에 거의 그대로 매핑됨. 그 CRUD/스키마 경험 재사용 가능.

**결정 필요**
- [ ] 트렌드 1건당 대표 콘텐츠 N개를 어떻게 수집·연결 (D1과 연동)
- [ ] 출처 표기로 신뢰 + 저작권/임베드 정책

---

## D7. 🟡 데이터 신선도 / 갱신 주기

**왜 차별점인가**: 트렌드는 실시간성이 생명 (Google Trends ~10분 갱신을 기준점으로 제시). 너무 느리면 "트렌드"가 아님.

**결정 필요**
- [ ] 갱신 주기 목표(예: 시간 단위) vs 수집·LLM 비용 트레이드오프
- [ ] "마지막 갱신 시각" 노출로 신뢰 확보

---

## D8. 🟢 소비자형 피드 UX (분석툴이 아니라 피드 앱)

**왜 차별점인가**: 썸트렌드/빅카인즈는 실무형 분석툴. trendDrop은 *들어오자마자 이해되는* 소비자 경험이어야 차별 (리서치 4번 차별점). 현재 프로토타입 UI 방향성은 이미 이쪽.

**결정 필요**
- [ ] 정보 밀도/카드 구조 — 지표 나열 최소화, 스토리형 카드
- [ ] 관심 키워드 저장/재방문 유도(리서치 P2) 도입 시점

1. **D1 데이터 소스**를 현실적으로 무엇부터 — 이게 안 정해지면 나머지 다 공중에 뜸
2. **D3+D6**(왜 뜨나 + 근거 콘텐츠)을 MVP에서 어떤 최소 형태로 증명할지 — 여기가 해자
3. **D2 급상승 산식**의 v1 정의 — 하드코딩 숫자를 설명 가능한 실값으로
